These are requirements for a walk planner app.

1. Walks should be able to be created, edited, and deleted.
2. Walks should have a name, and a location.
2a. Walks should have a beach toggle, yes or no.
3. The location for a walk is picked from a map.
4. The map location will be used to select the nearest WeatherUnderground station.
5. The nearest WeatherUnderground station will be used to get the weather forecast for the walk.
6. Location of the map should start at users current location.
7. The app shall use english units for temperature, wind speed, etc.
8. If the beach toggle is set to yes, the app should use the nearest ocean tide information and 
    show the next low tide time on the walk card.
9. Walks have a go or no go status.  Walks are no go unless they meet the following criteria:
    1. Feels like temperature is between settings min and max.
    2. Wind speed is below settings wind speed max.
    3. If it's a beach walk, the tide should be within settings beach tide window plus or minus of low tide.
    4. It should not be raining per settings.
10. Should have a Settings button on the bottom. 
11. Settings import/export.
    Should have buttons to export and import all data in JSON format for backup.
    Should have a button to delete all data.
    These buttons should be in Settings.
    These should be on the bottom of settings screen.
11a. Settings go/no go parameters.
    Should have feels like temperature min and max.
    Should have wind speed max.
    Should have beach tide window.
    Should have rain toggle.
12. Future walk forecast.
    Should show the walk go/no-go status for the next 7 days on the walk card.
    Go/no go should be evaluated for each hour of daylight.  If there is at least one hour of go, 
    the walk should be considered go for that day.
    Tide should not be considered for future walk forecasts.
13. Keep track of the last time weather data was fetched.  Only update it if all walks are updated.
14. Should have a version number that starts with 1.0.0.


Style
1. The initial view of the app should be the list of walk cards. The map should be hidden.  
    No walk editor should be displayed.
2. At the bottom of the walk list should be an add walk button.
3. The walk editor will appear when the add walk button is clicked or a walk is editted.
4. The walk editor will be full screen and include a map to select the walk location.
5. The walk editor will have a save button and a cancel button.
6. The save button will save the walk and close the editor.
7. The cancel button will close the editor without saving.
8. Current conditions will be displayed on each walk card and update every 60 minutes.
9. Show feels like temperature on the walk card.
10. Do not show edit and delete buttons on the walk card.
11. When a walk card is clicked, open the the walk information view.
    This view should have a button to open the editor.
    This view should show each daylight hour of the next 7 days with a go/no-go status in a compact form.
    For beach walks, include the tide in the determination for go/no-go status for each hour.
    Each daylight hour should be represented by a green or red pill with the hour, no need to show am/pm.
    After the pills, show a button "Detail".  Pressing this button should show the temperature, wind speed, and weather code for each hour.
    Each of the next 7 days should show when low tide occurs during daylight hours by just showing the time.
    This view should show the current conditions.
11a. When the editor opens zoom in on the walk location.  If it's a new walk, 
    use the current location of the user.
12. The editor should have a delete button.
13. When a walk is go, it should have a green background on the walk card, and be sorted 
    to the top of the list.  
14. When a walk is no go, it should have a light red background on the walk card.
15. Have a button above Settings that shows the last time weather data was fetched plus "[Update]".
    Pressing this button should update the weather data for all walks.
16. Should show the version number on the bottom of the settings screen.
17. PWA (Progressive Web App) Support:
    1. The app shall include a Web App Manifest to allow installation on home screens.
    2. The app shall use a Service Worker to cache HTML, CSS, JS, and map assets for offline use.
    3. The app shall specify theme colors and use the project icon for its home screen shortcut.
18. Push Notifications:
    1. The app shall implement the Web Push API.
    2. The app shall allow users to opt-in to notifications on their devices.
    3. The app shall provide a toggle in the Settings view to enable or disable walk-ready notifications.
    4. The app shall notify the user when a walk is a go for the first time each day.
